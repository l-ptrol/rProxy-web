# Документация: Внедрение SSL для IP-адресов (Let's Encrypt + acme.sh)

Данный документ описывает техническую реализацию и процесс получения валидных SSL-сертификатов для публичных IP-адресов в проекте `rProxy-desktop` (начиная с версии v1.5.6-go).

## 1. Введение
Традиционно Let's Encrypt выдавал сертификаты только на доменные имена. Однако с 2025 года (и особенно массово в 2026) CA Let's Encrypt поддерживает выдачу сертификатов на публичные IP-адреса через специальный профиль «Short-lived Certificates».

**Ключевое ограничение**: Сертификаты для IP имеют очень короткий срок жизни (обычно **160 часов** или около 6-7 дней), что требует полностью автоматизированного процесса обновления.

---

## 2. Инструментарий: Почему acme.sh?
Хотя Certbot является стандартом де-факто, его поддержка IP-сертификатов в некоторых версиях ограничена или требует специфических настроек.
В `rProxy-desktop` для работы с IP был выбран **acme.sh** по следующим причинам:
- Нативная и стабильная поддержка параметра `--certificate-profile shortlived`.
- Легковесность (написан на Shell) и отсутствие зависимостей от Python.
- Гибкая установка сертификатов в произвольные директории.

---

## 3. Этапы процесса внедрения

### Этап 1: Подготовка VPS
При нажатии кнопки «Настройка VPS» в интерфейсе rProxy, на удаленный сервер отправляется скрипт, который выполняет:
1. Установку необходимых зависимостей (`socat`, `curl`).
2. Установку `acme.sh` командой:
   ```bash
   curl https://get.acme.sh | sh -s email=admin@$(hostname -I | awk '{print $1}')
   ```
3. Регистрацию аккаунта в ACME-сервисе (используется Google Trust Services или Let's Encrypt).

### Этап 2: Конфигурация Nginx
Для прохождения проверки владения IP-адресом (HTTP-01 challenge), Nginx должен корректно отдавать токены.
В `rProxy-desktop` за это отвечает автоматическое добавление блока в конфигурации:
```nginx
location ~ ^/.well-known/(acme-challenge|pki-validation)/ {
    add_header Content-Type text/plain;
    root /var/www/letsencrypt;
}
```
*Примечание: acme.sh в режиме `--nginx` умеет временно модифицировать конфиг самостоятельно.*

### Этап 3: Выпуск сертификата (Issue)
При запросе SSL для IP, `rProxy-desktop` выполняет на VPS следующую команду:
```bash
~/.acme.sh/acme.sh --issue --nginx \
  -d [IP_ADDRESS] \
  --certificate-profile shortlived \
  --days 3 \
  --force
```
- `--nginx`: использование модуля Nginx для проверки.
- `--certificate-profile shortlived`: КРИТИЧЕСКИЙ флаг для IP-сертификатов.
- `--days 3`: принудительное обновление, если до истечения осталось менее 3 дней.

### Этап 4: Установка сертификата (Install)
После успешного выпуска сертификат копируется в системную директорию `/etc/nginx/ssl/` для обеспечения консистентности путей:
```bash
~/.acme.sh/acme.sh --install-cert \
  -d [IP_ADDRESS] \
  --key-file /etc/nginx/ssl/[IP].key \
  --fullchain-file /etc/nginx/ssl/[IP].crt \
  --reloadcmd 'systemctl reload nginx'
```

## 4. Схема работы в коде (v1.5.6-go)

### Логика переключения (vps.go)
Функция [RunCertbot](file:///D:/Python/antigravity/rProxy-desktop/core/vps.go#226-245) теперь выполняет роль диспетчера:
```go
if isIP {
    // Вызов команд acme.sh для IP
} else {
    // Вызов Certbot для обычных доменов
}
```

### Динамические пути (services.go)
Генератор конфигов Nginx [GenerateNginxConf](file:///D:/Python/antigravity/rProxy-desktop/core/services.go#21-62) адаптирует пути в зависимости от типа хоста:
- Для доменов: `/etc/letsencrypt/live/...` (Пути Certbot).
- Для IP: `/etc/nginx/ssl/...` (Пути acme.sh).

---

## 5. Автоматическое обновление
`acme.sh` при установке автоматически прописывает задание в `crontab`. Оно проверяет сертификат раз в сутки. Если до конца срока действия (160 часов) осталось мало времени, он будет перевыпущен и Nginx будет перезагружен.

---

## 6. Возможные проблемы и решения
1. **Ошибка "Private-Use IP"**: Выпуск сертификата на серые/локальные IP (192.168.x.x и т.д.) невозможен. IP должен быть публичным.
2. **Порт 80 занят**: Убедитесь, что Nginx запущен и корректно слушает 80 порт, иначе проверка ACME не пройдет.
3. **Отсутствие acme.sh**: Если команда не найдена, необходимо повторно выполнить «Настройку VPS».
